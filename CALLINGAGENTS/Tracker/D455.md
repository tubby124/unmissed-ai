# Onboarding policy: prompt is read-only until first 5 calls

## Origin (2026-05-05)

Brian-class drift starts the moment someone hand-edits a fresh client's prompt. Onboarding is the highest-risk window because the operator is most tempted to "polish" before the first call. A policy that forces all early tuning through DB columns prevents drift before it begins.

## Build

1. New column: `clients.prompt_unlocked_at timestamptz NULL`
2. Set `prompt_unlocked_at = now()` when `successful_call_count >= 5` (count by post-call webhook completion).
3. Block direct prompt edits via dashboard prompt editor when `prompt_unlocked_at IS NULL`. Show banner: "Prompt edits unlock after 5 successful calls. Until then, tune via Settings (knowledge, tone, voice, FAQs, examples)."
4. Override available via admin role: a "Force unlock" button with explicit confirmation. Logs to audit table.
5. Admin-side override threshold tunable: `ONBOARDING_PROMPT_LOCK_CALL_COUNT` env var (default: 5).

## Acceptance

- [ ] New trial client → cannot edit raw prompt until 5 successful calls logged
- [ ] All 19 settings cards remain editable during lock period (knowledge, tone, voice, FAQs, etc.)
- [ ] Admin override works + logs to audit
- [ ] After 5 calls, lock auto-releases

## Why this layer

Cheapest preventive policy, but also the lowest-leverage of the 5 — only catches the "fresh-client-tuned-too-early" failure mode. D453 + D454 are stronger because they catch drift introduced anytime, not just during onboarding.

## Dependency order

Independent. Can ship before or after the others. Most effective AFTER D453 (so admins have structured editing surfaces during the lock period).
