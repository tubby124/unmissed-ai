# D450 — Add `twilio_number` to `needsAgentSync` (one-line propagation fix)

> **What this fixes:** the documented gap from [[Architecture/Control-Plane-Mutation-Contract]] §7 risk #2. Admin-only field. When an admin sets or changes `twilio_number` via God Mode, the change is written to the DB but `updateAgent()` is not called and `clients.tools` is not rebuilt. The SMS tool's presence in `toolOverrides` on the next call reflects the stale state until `sms_enabled` is toggled (which does trigger sync) or a manual `syncClientTools()` runs.

> **Why low priority:** admin-only path. No active client has hit this in production. Documented gap, not an observed incident.

> **Why ship anyway:** one-line code change with a one-line test. Eliminates a documented inconsistency between the field's mutation class (`DB_PLUS_TOOLS`) and its actual sync triggering. Removes a future source of confusion when the next admin sets a Twilio number for a new client and SMS doesn't work until something else triggers sync.

## Goal

Admin sets `twilio_number` via `PATCH /api/dashboard/settings`. Settings PATCH detects this in `updates`, includes it in `needsAgentSync`, and triggers `updateAgent()` + `syncClientTools()` like every other tool-affecting field.

## Files

### Modified
- [src/app/api/dashboard/settings/route.ts](../../src/app/api/dashboard/settings/route.ts) — extend `needsAgentSync` boolean expression. Currently (per mutation contract):

  ```ts
  const needsAgentSync = (
    'system_prompt' in updates ||
    'forwarding_number' in updates ||
    'transfer_conditions' in updates ||
    'booking_enabled' in updates ||
    'agent_voice_id' in updates ||
    'knowledge_backend' in updates ||
    'sms_enabled' in updates
    // missing: 'twilio_number' in updates
  )
  ```

  Add `'twilio_number' in updates`. **Verify** `twilio_number` is already in the `select()` block at line ~419 (mutation contract claims it is — confirm before assuming).

### New
- Test in [src/lib/__tests__/settings-route-needs-agent-sync.test.ts](../../src/lib/__tests__/) — assert that a PATCH containing `twilio_number` triggers `needsAgentSync` path. Mock `updateAgent` and verify it was called.

## Acceptance criteria

1. Admin PATCHes `twilio_number` in isolation (no other fields). Server runs `updateAgent()` and `syncClientTools()`. Response: `ultravox_synced: true`.
2. SMS tool registration on the live agent matches `(sms_enabled === true && twilio_number !== null)` after the PATCH.
3. Existing test suite passes (no regression).
4. Mutation contract §7 risk #2 status updated to `RESOLVED (D450, YYYY-MM-DD)`.

## Risks

1. **Settings PATCH select() may not include `twilio_number`.** Mutation contract §2 row "twilio_number" claims it does ("confirmed it IS included"). Verify before assuming. If missing, this is a 2-line change, not 1.
2. **God Mode write path also exists** (admin-only direct write to `clients.twilio_number` outside the settings PATCH). That path won't be fixed by this change. Out of scope — file as a separate gap if it matters. The settings PATCH is the supported user-facing path.
