# D307 — Recompose Warning UX

✅ 2026-04-22 — shipped together with D305 frontend.

## Problem
Admin recompose button nuked the full system prompt and synced to Ultravox with no confirmation, no preview, no diff. Any hand edits to sections were silently overwritten. High-regret action with zero safety gate.

## Solution
Wrapped the existing admin recompose button in `RecomposeConfirmDialog` (new file). Flow:
1. User clicks "Recompose Prompt…" in `AdminRecomposePanel`
2. Dialog opens and immediately POSTs to `/api/dashboard/variables/preview` with `{ recompose: true }` (dry-run — no DB write, no Ultravox sync)
3. Dialog renders: char count before → after, `promptChanged` flag, 12K-limit warning, and section-level diff (added / removed / modified / unchanged)
4. User can switch between **Summary** / **Current** / **New** tabs to read both prompts
5. "Confirm recompose" button disabled when no change, preview failed, or over 12K chars
6. Only on confirm: POSTs to `/api/admin/recompose-client` (existing mutation endpoint)
7. Dialog close blocked during mutation

## Files
- `src/components/dashboard/settings/RecomposeConfirmDialog.tsx` — new
- `src/components/dashboard/settings/AdminRecomposePanel.tsx` — simplified, opens dialog instead of direct POST
- `src/app/api/dashboard/variables/preview/route.ts` — consumed (already existed from D305 backend)

## What's Safe Now vs Still Open
**Safe:**
- Admin recompose requires explicit preview-first confirm
- Char overflow blocked at UI level (backend also blocks via `validatePrompt()`)
- No-op recomposes blocked (can't confirm when `promptChanged === false`)

**Still open (out of scope):**
- Section-level diff is heading-match only. If a heading is renamed it reads as removed+added. Good enough as a safety signal.
- No tests added — dialog is UI-only, logic is thin
- Client-facing recompose button not yet shipped. When it is, it reuses the same dialog.
