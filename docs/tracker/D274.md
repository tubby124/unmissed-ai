# D274 — System prompt = template with named slots

## Summary
Replace the monolithic template string with a slot-based composition system. The prompt is assembled from named slot functions, each producing a string or empty string. Output must be byte-identical to current output for all golden test snapshots.

## Why
Current `INBOUND_TEMPLATE_BODY` is a single massive string with `{{VARIABLE}}` substitution followed by 10+ post-build string injections. This makes it impossible to:
- Update one section without risking adjacent sections
- Conditionally include/exclude sections based on config
- Let the UI target a specific section for editing
- Know which variables feed which section

Named slots solve all of these. Each slot is an independent function with clear inputs and outputs.

## Implementation

### 2a. Slot registry + slot builder functions
Create `src/lib/prompt-slots.ts` with one function per slot from D285 spec:
- `buildSafetyPreamble(ctx)`, `buildForbiddenActions(ctx)`, `buildVoiceNaturalness(ctx)`
- `buildGrammar(ctx)`, `buildIdentity(ctx)`, `buildToneAndStyle(ctx)`
- `buildGoal(ctx)`, `buildConversationFlow(ctx)`, `buildAfterHours(ctx)`
- `buildEscalationTransfer(ctx)`, `buildReturningCaller(ctx)`, `buildInlineExamples(ctx)`
- `buildCallHandlingMode(ctx)`, `buildFaqPairs(ctx)`, `buildKnowledgeBase(ctx)`
- `buildCalendarBooking(ctx)`, `buildSmsFollowup(ctx)`, `buildVipProtocol(ctx)`

Create `composePrompt(slots: string[])` that joins non-empty slots with `\n\n`.

### 2b. Section markers for all slots
Extend `prompt-sections.ts` to wrap ALL slots with `<!-- unmissed:SLOT_ID -->` markers.

### 2c. Shadow mode integration test
Test that old path and new path produce byte-equal output (after stripping markers).

## Dependencies
- **Requires:** D285 (sandwich spec — defines slot order and structure)
- **Enables:** D268 (minimal base), D265 (KB removal), D272 (constraint removal), D283 (variable visibility), D280 (UI composition)
- **This is the backbone** — every downstream architectural item depends on it
