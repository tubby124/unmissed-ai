# D302 — Preserve Niche Intake Fields During Onboarding

## Status: DONE (2026-03-31)

## Problem
During onboarding, niche-specific intake fields (niche_emergency, niche_clientType, niche_pricingModel, etc.) were consumed once by `buildSlotContext()` to generate the prompt but the raw values were **never saved to the DB**. This meant `regenerateSlot()` fell back to niche defaults and silently lost customizations.

## Fix
Two changes:

### 1. Provision route — save niche_* fields (src/app/api/provision/trial/route.ts)
After `toIntakePayload()`, extract all `niche_*` keys from the intake payload and merge them with any explicit `nicheCustomVariables` (TRIAGE_DEEP, etc.). Write the merged object to `clients.niche_custom_variables`.

### 2. Slot regenerator — spread niche_* fields back (src/lib/slot-regenerator.ts)
In `clientRowToIntake()`, spread all `niche_*` keys from `niche_custom_variables` back as top-level intake fields so `buildSlotContext()` can read them directly (e.g., `intake.niche_emergency`).

## Test Results

- All 461 tests pass (456 existing + 5 new)

## Files Changed
- `src/app/api/provision/trial/route.ts` — extract + merge niche_* fields into niche_custom_variables at client insert
- `src/lib/slot-regenerator.ts` — spread niche_* fields from niche_custom_variables to top-level intake keys
- `src/lib/__tests__/slot-regenerator.test.ts` — 5 new D302 round-trip tests + updated helpers

## Impact
- Zero risk to existing clients (additive JSONB write)
- New clients onboarded after this change will have full round-trip fidelity
- Unblocks D303 (variable edit API) and D278 (Agent Brain)
