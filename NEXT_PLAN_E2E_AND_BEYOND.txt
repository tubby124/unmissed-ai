    - Fallback: inject state via query string or custom header from tool URL

B3-F2. COMPLETED WEBHOOK — PERSIST FINAL CALL STATE
    - Currently: completed webhook does NOT read final call_state
    - Impact: we lose end-of-call counters (total slotAttempts, bookingAttempts, etc.)
    - Fix: in completed webhook, read X-Call-State from Ultravox call metadata
      (or from the last tool response headers) and write to call_logs.call_state
    - Enables: per-call analytics ("this call tried 3 slots before booking")
    - Effort: Low — add parseCallState() call + DB column already exists

B3-F3. SMS + TRANSFER ROUTES — ADD STATE USAGE
    - Headers flow through (CALL_STATE_PARAM is on both tool builders) but:
      - /api/webhook/[slug]/transfer-status — no parseCallState() / setStateUpdate()
      - SMS send route — no parseCallState() / setStateUpdate()
    - What to add: read state, set lastToolOutcome ('transferred' / 'transfer_error'
      / 'sms_sent' / 'sms_blocked' / 'sms_error'), write updated state back
    - Low priority — these tools are single-fire (no retry coaching needed yet)

B3-F4. fieldsCollected + escalationFlag — POPULATE FROM TOOLS
    - Both fields exist in CallState but no tool writes to them
    - fieldsCollected: should accumulate as intake tools gather name/phone/service
    - escalationFlag: should be set when caller asks for manager or agent detects urgency
    - Implementation: add to tool routes that collect fields (currently manual in prompt)
    - Medium effort — requires defining which fields each niche tracks

B3-F5. DASHBOARD VISIBILITY — CALL STATE DATA
    - No UI shows call state data today
    - Add to call detail view in admin dashboard:
      - Final workflowType, step, slotAttempts, bookingAttempts, knowledgeQueries
      - Timeline: "slot check #1 → no slots → slot check #2 → found → book → slot taken → book #2 → success"
    - Depends on B3-F2 (persisting final state)
    - Medium effort — new component + API endpoint

B3-F6. STEP COUNTER — PER-WORKFLOW ADVANCEMENT
    - step counter only increments on booking attempts today
    - Need per-workflowType step definitions:
      - intake: greeting → field collection → urgency check → handoff
      - triage: greeting → need assessment → routing → transfer/message
      - booking: greeting → slot search → selection → confirmation
      - support: greeting → query → answer → follow-up
    - Each tool call should advance step based on workflowType + action
    - Medium-high effort — requires defining state machines per workflow
