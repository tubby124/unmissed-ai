# D424 — Niche Field Registry

## Problem
Niche-specific fields are scattered: some in `niche_custom_variables` JSONB, some as direct client columns, some in `intake_json`, some mapped by `intake-transform.ts`. No single place defines the canonical field list for a niche. Adding a new PM field requires touching 3–4 files with no guard against drift.

## Fix
Create a `src/lib/niche-field-registry.ts` that exports per-niche field definitions:
- `key`: the intake key (e.g., `niche_petPolicy`)
- `label`: display label for settings UI
- `type`: 'text' | 'textarea' | 'chip' | 'tel'
- `dbTarget`: 'niche_custom_variables' | column name
- `promptSlot`: which slot reads it (for traceability)

PmConfigCard and the PM onboarding step should be driven by this registry rather than hardcoded fields.

## Files
- `src/lib/niche-field-registry.ts` — new file
- `src/components/dashboard/settings/PmConfigCard.tsx` — consume registry
- `src/app/onboard/steps/niches/property-management.tsx` — consume registry
