# D433 — Fix Lite plan mode mismatch (voicemail_replacement vs triage)

**Status:** open
**Priority:** P0 — DO THIS FIRST
**Filed:** 2026-04-06

## Problem

Lite plan ("Call Catcher") users get `triage` (AI Receptionist) in both trial AND live accounts. The plan entitlements define `defaultMode: 'voicemail_replacement'` but two separate overrides ignore this.

Core/Pro provision correctly. Identity (name, business name, niche defaults) is correct on all plans. Only the behavioral mode is wrong for Lite.

## Root Cause

Two overrides, both force Lite → triage:

**1. Trial** (`provision/trial/route.ts` line 106) — intentional, show best product:
```typescript
const trialUpgradedMode = rawCallHandlingMode === 'message_only' ? 'triage' : rawCallHandlingMode
```

**2. Live** (`stripe/create-public-checkout/route.ts` lines 193–196) — bug, meant to block full_service only:
```typescript
const checkoutCallHandlingMode = checkoutEntitlements.bookingEnabled
  ? intakeData.call_handling_mode
  : 'triage'  // ← fires for Lite because bookingEnabled=false
```

## Decision Needed First

**Option A** — Lite stays triage. Rename "Call Catcher" to reflect capped AI Receptionist. No code change.

**Option B** — Lite gets `voicemail_replacement`. Fix the gate. 4-line change.

## Fix (Option B)

`src/app/api/stripe/create-public-checkout/route.ts`:
```typescript
// Before:
const checkoutCallHandlingMode = checkoutEntitlements.bookingEnabled
  ? intakeData.call_handling_mode
  : 'triage'

// After:
const rawMode = (intakeData.call_handling_mode as string) || 'triage'
const checkoutCallHandlingMode = (!checkoutEntitlements.bookingEnabled && rawMode === 'full_service')
  ? 'triage'
  : rawMode
```

Then decide whether trial Lite users should also see voicemail (honest) or triage (best-foot-forward).
