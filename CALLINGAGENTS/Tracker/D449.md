# D449 — Per-field "Saved, but not live yet" warning chip

> **What this fixes:** the propagation-failure trust gap. Currently the only signal an owner gets when a settings PATCH succeeded in DB but didn't reach the live agent (legacy-prompt patcher no-op, `regenerateSlots` failure on monolithic prompts, `updateAgent()` 5xx) is the page-level amber legacy-prompt banner from [[Tracker/D369]] — and that's only one failure mode. D449 raises a per-field chip the moment a save doesn't propagate, regardless of failure category.

> **Independent of [[Tracker/D447]] and [[Tracker/D448]].** D447 makes runtime drift visible by polling Ultravox; D449 makes propagation failure visible at the moment of the PATCH. Different signal, different cost. D449 has no Ultravox API quota cost — fires entirely off `last_agent_sync_status` + `last_agent_sync_error` columns already populated on every PATCH.

## Goal

Each settings card field that triggers `needsAgentSync` shows an inline yellow chip when:
- the most recent PATCH for that field returned `ultravox_synced: false`, OR
- `regenerateSlots()` returned `success: false` on that field's recompose attempt, OR
- `last_agent_sync_status === 'error'` on the field's owning client row.

Chip copy: **"Saved, but not yet live on your agent. [Why?] [Retry]"**.

The "Why?" expand reveals the specific reason (legacy-prompt format → migrate; Ultravox 5xx → retry; plan-gated → upgrade). The "Retry" CTA re-triggers `needsAgentSync` for that field via a no-op PATCH.

## Scope

1. **Server response shape** — `PATCH /api/dashboard/settings` already returns `{ ok, ultravox_synced, ultravox_error?, warnings? }` per [[Architecture/Control-Plane-Mutation-Contract]] §3 step 8. Extend to include per-field sync status when multiple fields are in one PATCH (today the response is whole-PATCH; D449 needs field-resolution).
2. **`usePatchSettings` hook** — surface per-field sync status to consuming components. Today it serializes PATCHes per-client; need to cache the last-known sync status per field key.
3. **Per-card chip rendering** — every card that owns a `DB_PLUS_PROMPT`, `DB_PLUS_TOOLS`, or `DB_PLUS_PROMPT_PLUS_TOOLS` field renders the chip when its field has a stale-sync status.

## Acceptance criteria

1. Owner edits Greeting on a snowflake client. PATCH returns `ultravox_synced: true` (DB updated) but `field_sync_status.GREETING_LINE === 'error'` because `regenerateSlots` no-op'd on the legacy-monolithic prompt. Chip appears under the field with `legacy_prompt_patcher_noop` reason and a "Migrate" CTA linking to the [[Tracker/D445]] migration flow.
2. Owner edits `forwarding_number` while Ultravox is reachable. Chip never appears. PATCH returns success.
3. Owner edits `forwarding_number` while Ultravox is 503'ing. Chip appears with `ultravox_5xx` reason. "Retry" button issues a no-op PATCH to re-trigger `needsAgentSync`. Chip clears on success.
4. Owner edits a registry-readonly variable (D443 has already hidden the Edit button — this case shouldn't reach the PATCH route). If somehow a PATCH does fire on a readonly field, the chip shows `fake_control` reason. Belt-and-suspenders only.
5. The page-level amber banner from D369 still renders. After D449 ships and 30 days of no negative feedback, file a follow-up to remove D369 banner from Settings (Overview page may still want it).

## Risks

1. **Field-level sync resolution requires extending the settings PATCH response.** Today it's whole-PATCH. Splitting per-field needs care — some fields share a single agent sync (e.g., `forwarding_number` + `transfer_conditions` both ride one `updateAgent()` call). Resolution: if `needsAgentSync` fires once for a multi-field PATCH and fails, ALL fields in that PATCH get the same `error` status. Acceptable.
2. **"Retry" CTA re-PATCH could loop** if Ultravox is hard-down. Mitigation: client-side retry budget (max 3 retries per field per page load), then chip becomes "Persistent failure — contact support".
3. **Chip clutter on multi-field cards.** If 4 fields on the SMS card all show chips, the card looks broken. Mitigation: collapse multiple chips on the same card into one summary chip ("3 fields not yet live") with a click-to-expand list.
