## Three-Concern Separation

Property management is NOT "just RAG." It requires three distinct data layers:

| Concern | Layer | Implementation | Status |
|---------|-------|---------------|--------|
| **A. Structured Records (READ)** | Tenant table embedded in prompt via contextData | `context-data.ts` → `buildContextBlock()` | DONE (pre-refactor) |
| **B. Retrieval (SEARCH)** | Building policies, PM procedures, tenant FAQs via queryCorpus | `knowledge-retrieval.ts` → gated on useKnowledgeLookup | DONE (Phase 4) |
| **C. Write Actions** | Maintenance request creation, note appending | `pm-ops.ts` → types, validation, capability gating | DONE (Phase 7) |

## What the Voice Agent Can Do

### READ (always available for PM niche)
- Look up tenant by unit number, address, or name (contextData table in prompt)
- Confirm tenant identity ("I have unit 4705B under John Smith — is that right?")
- Answer policy/procedure questions from knowledge base (queryCorpus)

### WRITE (gated on `updateTenantRequests=true`)
- **Create maintenance request** — collects unit, tenant, category, description, urgency, access window, entry permission
- **Append note** to existing request

### CANNOT DO (bounded)
- Cannot update request status (dashboard/API only)
- Cannot delete or cancel requests
- Cannot reassign or schedule technicians
- Cannot confirm repair timelines or costs
- Cannot create/modify tenant records
- Cannot process payments or billing
- Cannot commit to specific appointment times

## Data Model

### MaintenanceRequest

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| id | UUID | auto | Generated by system |
| clientId | string | yes | FK to clients table |
| unitNumber | string | yes | Primary lookup key |
| tenantName | string | yes | From tenant table or caller |
| callerPhone | string | no | From callerContext (never asked) |
| category | MaintenanceCategory | yes | plumbing/hvac/electrical/appliance/structural/pest/lockout/other |
| description | string (10+ chars) | yes | Caller's issue description |
| urgencyTier | 'urgent' or 'routine' | yes | emergency_911 NEVER creates a request |
| preferredAccessWindow | string | no | "Tuesday afternoon" etc |
| entryPermission | boolean | no | Can team enter if not home? |
| status | RequestStatus | auto | Always 'new' when voice agent creates |
| createdAt | ISO string | auto | Timestamp |
| createdBy | CreatedBySource | auto | Always 'voice_agent' for calls |
| callLogId | string | no | FK to call_logs for traceability |
| notes | string[] | auto | Starts empty, append-only |

### Urgency Tiers

| Tier | Voice Agent Action | Request Created? |
|------|-------------------|-----------------|
| emergency_911 | "Call 9-1-1 right now" → hangUp | NO — never a request |
| urgent | Flag for same-day PM callback + SMS confirmation | YES |
| routine | Log request + SMS confirmation | YES |

### Maintenance Categories

`plumbing` · `hvac` · `electrical` · `appliance` · `structural` · `pest` · `lockout` · `other`

## Capability Gating

| Capability Flag | Who Has It | What It Gates |
|----------------|-----------|---------------|
| `useTenantLookup` | property_management only | Tenant table lookup, existing request lookup |
| `updateTenantRequests` | property_management only | createMaintenanceRequest, appendRequestNote |
| `emergencyRouting` | property_management, hvac, plumbing | Niche-specific emergency triage in TRIAGE_DEEP |

No other niche can access PM write operations. Cross-niche safety is enforced by:
1. `isWriteActionAllowed(capabilities)` — returns false for all non-PM niches
2. `getAllowedPmOps(capabilities)` — returns empty for niches without tenant lookup
3. 56 tests verify these boundaries

## Urgency Classification (Heuristic)

`classifyUrgency(description)` provides a keyword-based suggestion:

| Keywords | Classification |
|----------|---------------|
| gas smell/leak, fire, CO detector, break-in | emergency_911 |
| no heat, burst pipe, sewage, water heater, no electricity, flooding | urgent |
| Everything else | routine (safe default) |

This is for validation/logging. The voice agent's TRIAGE_DEEP prompt section handles actual call routing.

## How Data Flows (Current Architecture)

```
Caller → Twilio → inbound/route.ts
                      ↓
              buildAgentContext()
                      ↓
              ┌───────────────────┐
              │ A. contextData    │ → tenant table in prompt (READ)
              │ B. queryCorpus    │ → policies/docs retrieval (SEARCH)
              │ C. pm-ops.ts      │ → validation for future tool endpoint (WRITE)
              └───────────────────┘
```

### Current state:
- A (READ) and B (SEARCH) are live and wired
- C (WRITE) types and validation exist in `pm-ops.ts`
- The actual Ultravox HTTP tool endpoint for write-back is NOT yet built
- When built, it will call `validateMaintenanceRequest()` before persisting

### When a PM client with write-back onboards:
1. Create Supabase `maintenance_requests` table matching the TypeScript type
2. Create `/api/webhook/[slug]/maintenance` endpoint that validates via `pm-ops.ts`
3. Register as Ultravox HTTP tool (temporaryTool) gated on `updateTenantRequests`
4. Add tool to `buildAgentContext()` tool list for PM niche only
