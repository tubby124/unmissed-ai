# D277 — Lag Root Cause Investigation (plumber-calgary-nw)

## Problem
plumber-calgary-nw agent was lagging badly. Original prompt was 18.5K chars (way over 12K hard max). But user noted other 20K+ systems don't lag — so prompt size alone may not be the cause.

## Investigation Plan
- Compare tool count between lagging agent and working agents
- Check knowledge_backend config (pgvector with 20 chunks but 0 hits)
- Check if too many tools registered (knowledge tool + unused tools)
- Compare voice_id latency
- Check if contextData injection is adding significant size at call time
- Test after 5.3K compressed prompt is synced to Ultravox

## Ultravox Research Findings (2026-03-31)

Background research confirmed the following relevant to this investigation:

| Finding | Implication for D277 |
|---------|---------------------|
| No hard token limit in Ultravox | Lag is not from a hard cutoff but from responsiveness degradation |
| Recommended: under 6K chars for voice | 18.5K was 3x the recommended ceiling — compression to 5.3K is within range |
| queryKnowledge for all facts | Knowledge tool registered with 0 hits may cause silent failed queries adding latency |
| Monoprompt + tools is sufficient | Too many tools registered could contribute to tool selection overhead |
| contextData for structured data | Duplicate services in contextData add unnecessary tokens at call time |
| Don't use Ultravox Corpus (2 limit) | Not relevant to lag, but confirms pgvector is correct approach |

### Likely contributing factors (ranked)
1. **Prompt size (18.5K chars)** — 3x recommended voice agent size. Primary suspect. Now compressed to 5.3K.
2. **Knowledge tool with 0 hits** — Agent may be attempting knowledge queries that return nothing, adding round-trip latency per query attempt.
3. **Tool count** — Needs comparison with working agents. More tools = more tokens in tool descriptions = slower initial processing.
4. **Duplicate services in context_data** — 21 services (with dupes) injected at call time adds unnecessary token overhead.
5. **GLM-4.6 attention behavior** — Possible attention degradation at certain token counts (unconfirmed).
