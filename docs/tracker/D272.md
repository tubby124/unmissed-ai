# D272 — Remove business-logic constraints from prompts

## Summary
Audit FORBIDDEN_ACTIONS section. Keep only ~5 true safety rules hardcoded. Move all business-logic constraints (like "NEVER quote prices") to conditional rules based on client config.

## Why
The current FORBIDDEN_ACTIONS has 16 rules. Many are business-logic that override user-provided data:
- Rule 3: "NEVER quote specific prices" — contradicts clients who entered pricing in their FAQ/knowledge
- Rule 5: "NEVER say you are transferring unless transfer is enabled" — should be conditional on `TRANSFER_ENABLED`
- Rule 12: phone number handling — context-dependent

This violates the "User Designs the Prompt" ADR. If the user put pricing data in, the agent should use it.

## Classification of Current Rules
**KEEP (true safety):** Rules 1, 2, 8, 10, 11, 14, 15, 16 — formatting, prompt injection defense, hangUp safety
**CONDITIONAL (business logic):**
- Rule 3 (prices) → depends on `pricing_policy` field: 'quote_from_kb' / 'share_range' = OMIT rule; 'never_quote' / empty = KEEP
- Rule 5 (transfer) → depends on `TRANSFER_ENABLED`
- Rule 12 (phone) → context-dependent
**KEEP but soften:** Rules 4, 6, 7, 9, 13 — conversation flow rules, allow override via config

## Implementation
- Audit each rule in `template-body.ts` FORBIDDEN_ACTIONS
- Move conditional rules to `buildForbiddenActions()` in `prompt-slots.ts`
- Static safety rules stay hardcoded (~5 rules, ~500 chars)
- Business logic rules become conditional on DB fields
- May need new `pricing_policy` field on `clients` table

## Dependencies
- **Requires:** D274 (named slots for clean separation)
- **Pairs with:** D265 (removing KB bloat), D268 (minimal base)
- **Enables:** D280 (UI-driven composition — users control their own constraints)
- **Related ADR:** [[User Designs Prompt]]

## Key Files
- `src/lib/prompt-config/template-body.ts` — trim FORBIDDEN_ACTIONS
- `src/lib/prompt-slots.ts` — buildForbiddenActions() with conditional logic
- Possibly `src/lib/prompt-config/safety-rules.ts` — new file for clean separation

## Risk
HIGH — changes safety-critical prompt section. Must test with adversarial inputs. Mitigated by keeping all true safety rules (prompt injection defense, 911 override) untouched.
