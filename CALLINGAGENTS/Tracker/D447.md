# D447 — Show runtime truth on Overview, not stored DB truth

> **Why this leads:** This is Brian's literal complaint ("if I change the greeting here, will it work there?") and applies to **every** drift class: fake-control, propagation, partial-failure, intentional-gap. If Overview tiles render what is actually deployed, every drift becomes visibly diagnosable to the owner without a per-symptom backend fix. Single change, universal impact. See [[Tracker/D442]] audit recommended fix order. Per [[Architecture/Control-Plane-Mutation-Contract]] §7, current Overview leaks 5+ documented intentional gaps + 3 known silent-failure categories — all collapse into one pattern when the UI binds to runtime truth.

> **Pairs with — but does not depend on — [[Tracker/D445]] forceRecompose bypass (shipped 2026-04-30).** D447 makes drift visible; D445 fixes it at the prompt layer. They are independent. D447 ships value even if no snowflake is ever migrated.

## Goal

Replace each Overview tile's display source from `clients.{column}` (stored DB truth) to the deployed-agent runtime state, with an explicit divergence indicator when DB and runtime disagree.

Owner mental model after this ships:

> "What I see on Overview = what the agent actually says/does on the next call."

If a value can't reach the agent (registry-readonly variable, legacy-prompt patcher no-op, plan-gated tool, medium mismatch), the tile **shows the runtime value, not the DB value**, and a per-tile chip explains why.

## Scope

Three things, in order:

1. **New endpoint** — `GET /api/dashboard/agent/runtime-state` returning the resolved runtime state for the requesting client (auth-gated like other dashboard endpoints).
2. **Refactor 3 Overview components** to consume the new endpoint instead of reading directly from the dashboard `home` payload.
3. **Per-tile divergence chip** rendered when DB and runtime disagree on a specific field (greeting, voice, tools registered, capability flag).

Out of scope (separate tickets):
- Per-field "Saved, but not live yet" warning chip on PATCH — that is **Fix 2** (open as **D448** after D447 ships, or fold into [[Tracker/D369]] — the legacy-prompt banner — which already lives in `SettingsView.tsx`).

## Files

### New
- [src/app/api/dashboard/agent/runtime-state/route.ts](../../src/app/api/dashboard/agent/runtime-state/route.ts) — single GET. Returns:

  ```ts
  {
    deployed: {
      greeting: string,           // extracted from Ultravox callTemplate.systemPrompt
      voiceId: string,
      tools: string[],            // selectedTools modelToolName[] from callTemplate
      systemPromptCharCount: number,
      lastSyncedAt: string | null
    },
    db: {                          // for per-field comparison
      greeting: string,
      voiceId: string,
      tools: string[],            // from clients.tools (the override-tools source)
      systemPromptCharCount: number
    },
    syncStatus: 'success' | 'error' | 'unknown',  // from clients.last_agent_sync_status
    syncError: string | null,                      // from clients.last_agent_sync_error
    divergence: Array<{
      field: 'greeting' | 'voice' | 'tools' | 'systemPrompt',
      reason: 'fake_control' | 'propagation_failure' | 'partial_failure' | 'medium_constraint' | 'plan_gated' | 'unknown',
      dbValue: string,
      runtimeValue: string,
      cta?: { label: string, href: string }
    }>
  }
  ```

  Auth: `client_users` lookup (owner or admin).
  Cache: server-side LRU keyed by `clientId`, 60s TTL — single Ultravox `GET /api/agents/{id}` per minute per client per process. Stale-while-revalidate on 401/timeout from Ultravox.
  Fail mode: if Ultravox unreachable, return DB values + `syncStatus: 'unknown'` + empty divergence array. Never block the Overview render.

## Ultravox API budget — non-trivial

Each Overview load = 1 GET against Ultravox. Without caching:
- 5 active clients × N owner sessions × M Overview re-mounts/min = unbounded multiplier on Ultravox quota.

**Mitigation (must ship with the route):**
- 60s server-side LRU per `clientId`. Two consecutive Overview loads within 60s = 1 upstream call.
- ETag on the response. Client revalidates every 30s but server returns 304 if cached entry is fresh.
- Bypass cache on a `?fresh=1` query param the divergence chip's "Refresh" button can use — owner-triggered, never automatic.
- Hard rate limit: max 1 upstream Ultravox GET per `clientId` per 30s, queued behind the LRU. If two Overview tabs race, second one waits on the first's cached result.

**Open question:** is there a meaningfully cheaper endpoint than `/agents/{id}` to get just the deployed prompt + tools? If Ultravox exposes a "lightweight agent metadata" endpoint, prefer it. Investigate before implementing — if not, full GET is fine at 60s cache.

## Divergence reason classification

The route is responsible for classifying *why* DB and runtime disagree. This is what makes the chip useful instead of just noise. Mapping:

| `reason` | Trigger condition | Example user-facing chip |
|---|---|---|
| `fake_control` | Variable is `editable:false` in registry, but DB and runtime differ (shouldn't happen post-D443; reported as bug) | "This field is computed — contact support" |
| `propagation_failure` | `last_agent_sync_status='error'` OR `regenerateSlots` returned `success:false` | "Saved but not yet live — your prompt format is legacy. [Migrate]" |
| `partial_failure` | DB write succeeded, `updateAgent()` ran, but Ultravox response shows no change | "Ultravox didn't accept the change. Retry?" |
| `medium_constraint` | DB toggle on, runtime tool present, but tool requires PSTN (transfer/IVR) — no actual divergence, but warn for browser test calls | already shipped via Fix 4 footnote — no chip needed |
| `plan_gated` | DB toggle on, runtime tool absent because plan doesn't include capability | "Your current plan doesn't include this. [Upgrade]" |
| `unknown` | Catch-all | "DB and live agent disagree. [Refresh]" |

The classifier lives server-side in the new route. UI just renders the strings.

## Acceptance criteria

1. Overview Greeting tile renders the value the agent actually says on the next call.
2. For Brian's `calgary-property-leasing` (slot pipeline, post-D443) and the 4 snowflakes (legacy-monolithic, pre-D445):
   - Greeting tile shows what the deployed prompt actually contains, not what `niche_custom_variables.GREETING_LINE` says.
   - If they disagree (snowflake case), tile shows the runtime value with a `propagation_failure` chip.
   - If they agree (slot-pipeline case post-D305), no chip.
3. Capability pills (QuickConfigStrip + CapabilitiesCard) reflect `clients.tools` AND deployed `selectedTools`. Pills with DB-on-but-tool-absent show "Setup incomplete" + deep link.
4. Overview load adds ≤1 Ultravox API call per `clientId` per 60s under steady-state load.
5. If Ultravox is down, Overview still renders (DB values, `syncStatus: 'unknown'`, no divergence chips).
6. New tests in `src/lib/__tests__/`:
   - Pure unit test for the divergence classifier (5+ cases per `reason`).
   - Route-level test asserting cache behavior (mocked Ultravox client; second call within 60s does not hit upstream).

## Risks

1. **Ultravox quota.** New steady-state cost. Mitigated by the 60s LRU + ETag + per-clientId rate limit. Monitor on first deploy with one client (Hasan), expand to all 5 only after metrics confirm.
2. **Greeting extraction is fragile.** "Deployed greeting" requires parsing `callTemplate.systemPrompt` for the greeting line. Slot-pipeline prompts have a stable `<!-- unmissed:identity -->` marker; legacy-monolithic prompts have hand-named headers. Implementation should: (a) use slot markers when present, (b) fall back to a regex over `<!-- unmissed:greeting -->` or `# GREETING` style headers, (c) fall back to first 3 sentences if neither found. Surface "could not extract" gracefully — render "Greeting on file" placeholder instead of garbage.
3. **Divergence false-positives.** Whitespace, marker stripping (`stripPromptMarkers()` in updateAgent), and `{{templateContext}}` placeholder handling all cause cosmetic diffs that aren't real drift. Normalize both sides identically (strip markers, collapse whitespace, leave placeholders as-is) before comparing — same approach the [drift-detector agent rule](../../.claude/rules/drift-detection-pattern.md) already documents.
4. **Tool list comparison ordering.** `clients.tools` and `callTemplate.selectedTools` may be ordered differently. Sort by `modelToolName` || `nameOverride` before comparing.
5. **Brian-style false hope.** This fix surfaces drift, it does not eliminate it. Owners who edit a registry-readonly variable will now see a `fake_control` chip — not the change reflected. Pair with [[Tracker/D443]] (already shipped) for the `editable:false` hide-Edit-button gate so the failure mode never reaches the chip in the first place.
