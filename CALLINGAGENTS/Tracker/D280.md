# D280 — UI-Driven Prompt Composition (The End State)

## Vision
The user IS the prompt builder. They don't know it — but every field they fill in (FAQs, prices, services, caller reasons, hours, emergency words, tone preference) becomes a template variable injected into the system prompt.

## Current Problem
- System prompt has hardcoded rules ("NEVER quote prices") that override user-provided data
- Template body has hardcoded personality ("energetic, capable, efficient") — D275 partially fixes this
- Niche defaults add constraints the user never asked for
- User can't see or control what's in their prompt without raw text editing
- The prompt is treated as a static artifact instead of a derived view of user inputs

## Architecture
```
User Inputs (UI)
  ├─ Identity: agent_name, business_name, close_person → {{AGENT_NAME}}, {{BUSINESS_NAME}}, etc.
  ├─ Personality: voice_style_preset → {{PERSONALITY_LINE}}, {{TONE_STYLE_BLOCK}}
  ├─ Services: service_catalog → {{SERVICES_OFFERED}}
  ├─ Routing: caller_reasons → {{TRIAGE_DEEP}}
  ├─ Knowledge: FAQs, pricing, policies → {{contextData}}, {{businessFacts}}
  ├─ Flow: booking_enabled, sms_enabled → conditional sections
  └─ Safety: hard constraints (non-negotiable, stay always)
        ↓
  buildPromptFromIntake() → DERIVED system_prompt
        ↓
  Ultravox agent (live)
```

## Key Principles
1. User's data is authoritative — never override with hardcoded defaults
2. Hard constraints = safety only (no impersonation, no medical advice, hang-up rules)
3. Everything else = user-configurable via template variables
4. Dashboard shows "what your agent knows" — every variable visible and editable
5. No user ever needs to see raw prompt text

### Known limitation
recomposePrompt() overwrites D251 manual section edits that live only in prompt text. Always use dryRun=true first. For targeted updates, use regenerateSlots() which only touches affected sections.
