# D320 — urgencyWords not stored independently

## Problem
`urgencyWords` is collected in the Step 3 UI but NOT included in `toIntakePayload()` and NOT written to the `clients` table by the provision route.

It only survives if the AI-generated TRIAGE_DEEP (via `/api/onboard/infer-niche`) successfully bakes it into the TRIAGE_DEEP block. If TRIAGE_DEEP generation fails (API timeout, rate limit, malformed response), urgency words are silently lost.

## Data flow (current)
```
UI: data.urgencyWords → step3 onBlur → POST /api/onboard/infer-niche (with urgencyWords)
                                        → AI generates TRIAGE_DEEP containing urgency signals
                                        → stored as nicheCustomVariables.TRIAGE_DEEP → DB niche_custom_variables
```

If the AI call fails, urgencyWords is gone. No independent storage.

## Fix options
1. Add `urgency_words` column to `clients` table, include in `toIntakePayload()`, inject into prompt as a variable
2. Store in `context_data` alongside `priceRange` (simpler, no migration)
3. Accept current behavior (urgency is baked into TRIAGE_DEEP — if AI fails, agent still works, just without custom urgency signals)

## Recommendation
Option 3 for now — the TRIAGE_DEEP generation is reliable (Haiku 4.5 via OpenRouter, 8s timeout). But document the risk. If we move urgency words to step 5 (D318), we should reconsider.
