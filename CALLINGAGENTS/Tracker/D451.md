# Hard char-delta gate on recompose

## Origin (2026-05-05)

After PR #70 merged, recomposing Brian (calgary-property-leasing) to pick up the returning-caller fix would have silently dropped 3,392 chars of hand-tuned custom content (Examples A-H — Saskatoon-specific triage, gas-leak handling, severity-first flow). Detected only because we ran a local script with section diffing — the `RecomposeConfirmDialog` would have shown a diff but had no hard gate against accepting massive content loss.

## Build

1. Add char-delta computation to [`/api/dashboard/variables/preview`](src/app/api/dashboard/variables/preview/route.ts) response: `{ charsDropped, charsAdded, percentChange }`
2. In [`RecomposeConfirmDialog`](src/components/dashboard/settings/RecomposeConfirmDialog.tsx):
   - If `charsDropped > 500` OR `percentChange > 5%`: confirm button disabled until "I accept content loss" checkbox is ticked
   - Show top-3 sections with biggest drop in the diff panel, before the unchanged ones
   - Banner above dialog: "This recompose will permanently remove 3,392 chars of stored prompt content. Custom edits not backed by DB columns will be lost."
3. Threshold tunable via `RECOMPOSE_DELTA_THRESHOLD_CHARS` and `RECOMPOSE_DELTA_THRESHOLD_PCT` env vars. Defaults: 500 chars / 5%.

## Acceptance

- [ ] Dryrun on Brian shows banner + disabled confirm + checkbox requirement
- [ ] Dryrun on a clean slot-pipeline client (delta < 50 chars) shows no banner, confirm enabled
- [ ] Test added to `RecomposeConfirmDialog.test.tsx` for both threshold paths
- [ ] Banner copy reviewed against fake-control rules in `core-operating-mode.md`

## Why this layer first

Cheapest of the 5 prevention layers (small dialog change, no DB / API surface). Stops the next Brian-class recompose from going through silently while D453 (promote-on-edit) is still being designed.
