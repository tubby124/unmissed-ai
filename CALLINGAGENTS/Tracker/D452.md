# Weekly drift detection cron

## Origin (2026-05-05)

Brian's prompt diverged from his slot-pipeline output by 3,392 chars and we had no idea until we manually ran a recompose dryrun. Same risk applies silently to every "slot-pipeline" client — anyone who's been hand-tuned via dashboard editor, `/build-prompt-ui`, or direct SQL has unknown drift exposure. We currently learn about drift only when a slot-code change forces a recompose, which is the worst possible time to discover content loss.

## Build

1. New table `client_drift_log`:
   ```sql
   CREATE TABLE client_drift_log (
     id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
     client_id uuid NOT NULL REFERENCES clients(id) ON DELETE CASCADE,
     checked_at timestamptz NOT NULL DEFAULT now(),
     stored_chars int NOT NULL,
     recomposed_chars int NOT NULL,
     chars_dropped int NOT NULL,
     chars_added int NOT NULL,
     pct_change numeric(5,2) NOT NULL,
     biggest_drop_section text,
     diff_summary text
   );
   CREATE INDEX ON client_drift_log (client_id, checked_at DESC);
   ```
2. Cron: `scripts/drift-check-all.ts` — for each active client, run `recomposePrompt(dryRun=true, force=false)`, compute deltas, insert row. Schedule weekly via Railway cron or Supabase pg_cron.
3. Admin dashboard surface: new section "Drift watch" on `/dashboard/admin` showing top 10 most-drifted clients with last check date + drift severity (green <100 chars, amber <500, red >500).
4. Per-client chip on `/dashboard/clients` table: "Drift: 3.4K chars" red badge for high-drift clients.

## Acceptance

- [ ] Cron runs weekly without timeout (10+ clients)
- [ ] Drift log populated; query `SELECT * FROM client_drift_log WHERE chars_dropped > 500 ORDER BY checked_at DESC` returns Brian + any others
- [ ] Admin surface shows correct counts
- [ ] Per-client chip clickable → opens drift detail modal with section diff
