# D405 — Maintenance request write-back: API + DB table

**Status:** 🔴 Open
**Priority:** HIGH
**Date:** 2026-04-05

## Problem
`pm-ops.ts` defines a complete data model + validation (`validateMaintenanceRequest`, `buildMaintenanceRequestPayload`) for maintenance requests — but there is NO API endpoint and NO database table. When Jade collects a maintenance request from a tenant, the information goes nowhere. The PM can only see it in the call transcript. This is the #1 gap between "AI voicemail" and "property management tool."

## Solution

### A. Supabase migration
```sql
create table maintenance_requests (
  id uuid primary key default gen_random_uuid(),
  client_id uuid references clients(id) on delete cascade,
  unit_number text not null,
  tenant_name text not null,
  caller_phone text,
  category text not null,
  description text not null,
  urgency_tier text not null check (urgency_tier in ('urgent', 'routine')),
  preferred_access_window text,
  entry_permission boolean,
  status text not null default 'new',
  created_at timestamptz default now(),
  created_by text not null default 'voice_agent',
  call_log_id uuid,
  notes text[] default '{}'
);
alter table maintenance_requests enable row level security;
create policy "clients own their requests" on maintenance_requests
  for all using (client_id = auth.uid());
```

### B. API route: `src/app/api/pm/maintenance-requests/route.ts`
- POST: validate with `validateMaintenanceRequest()` from `pm-ops.ts` → insert to DB → return `{id}`
- GET: return requests for authenticated client, filterable by `status` and `urgency_tier`
- Auth: require valid client session (same pattern as other routes)

### C. Register Ultravox HTTP tool in `agent-context.ts`
When `niche === 'property_management' && caps.updateTenantRequests`:
- Add `createMaintenanceRequest` HTTP tool pointing at the POST route
- Schema: `unit_number`, `tenant_name`, `category`, `description`, `urgency_tier`, `preferred_access_window`, `entry_permission`
