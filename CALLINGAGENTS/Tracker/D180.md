# D180 — message_only mode: TRIAGE overrides mode instruction

## Problem

`buildConversationFlow()` (Slot 8) generates a full TRIAGE block for every client. For `message_only` and `voicemail_replacement` modes, the `CALL_HANDLING_MODE` override (Slot 13) was supposed to suppress this. But GLM-4 / Llama treat earlier instructions as higher priority in long prompts (recency bias is weaker than expected — Sonar Pro confirmed). So TRIAGE was effectively running even for message-taking-only clients.

## Root Cause

No guard in `buildConversationFlow()`. The TRIAGE block was emitted unconditionally at Slot 8 and relied on Slot 13 to override it — which doesn't reliably work in long prompts.

## Fix

Added guard at the top of `buildConversationFlow()` in `src/lib/prompt-slots.ts`. For `message_only` and `voicemail_replacement` modes, the function now returns a simplified "collect name + phone + reason" flow immediately — TRIAGE section is never generated.

**Sonar Pro finding (2026-04-03):** Recency bias exists but is not sufficient to suppress a detailed earlier workflow section. "Place mode/goal last" is one mitigation; "don't emit the section at all" is cleaner and more reliable.
