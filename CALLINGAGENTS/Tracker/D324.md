# D324 — Plan = Mode (derive agent mode from plan selection)

## Problem
Users pick a "mode" (voicemail/lead capture/info hub/booking) AND a "plan" (Lite/Core/Pro) separately. These are redundant — the plan already describes what the agent does.

## Fix
Remove the mode selector from onboarding. Derive `call_handling_mode` from `selected_plan`:

| Plan | Mode | Behavior |
|------|------|----------|
| Lite ($49) | `voicemail_replacement` | Greet → collect message → text you |
| Core ($119) | `lead_capture` | Greet → triage → answer from KB → collect info → text you |
| Pro ($229) | `appointment_booking` | Everything Core does + calendar booking + live transfer |
| Trial | `lead_capture` (all features unlocked) | Full demo of all capabilities |

## Implementation
1. In `provision/trial/route.ts`: derive `call_handling_mode` from `selected_plan` instead of `data.agentMode`
2. In `activate-client.ts` (Stripe upgrade): update `call_handling_mode` when plan changes
3. Remove `agentMode` from `OnboardingData` type (or keep as optional override)
4. Remove step3-capabilities.tsx from step registry
5. Add mode derivation to `syncClientTools()` path on plan change

## Risk
- Existing clients have `call_handling_mode` set from onboarding — no migration needed, their mode stays
- New clients get mode from plan — clean path
- Dashboard could still expose a mode override for power users (future)
