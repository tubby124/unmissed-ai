# D299 — Collective Niche Intelligence (system gets smarter per niche)

## The Insight
Every client in a niche makes the next client's agent better. After onboarding 4 dentists, the 5th dentist gets battle-tested defaults instead of hand-crafted guesses.

## Current State
- `NICHE_DEFAULTS` in niche-defaults.ts — hand-crafted, static, never updated from production data
- `buildNicheFaqDefaults()` — hand-written FAQ per niche, never validated against real calls
- `knowledge_query_log` — tracks every question callers ask, per client, per niche (D252)
- `call_insights` — call classification, field collection rates, satisfaction proxies
- `knowledge_chunks` — approved FAQ/facts per client

All the raw data exists. No aggregation layer exists.

## Vision

### Niche Intelligence Registry
New table or view: `niche_intelligence`

```sql
-- Aggregated from all clients in a niche
niche TEXT,
top_questions JSONB,        -- top 20 questions callers ask (from knowledge_query_log)
proven_faq JSONB,           -- FAQ answers with highest caller satisfaction (from knowledge_chunks + call_insights)
best_triage TEXT,           -- triage flow that produces highest completion rate
common_objections JSONB,    -- objections detected across clients
avg_completion_rate FLOAT,  -- niche benchmark
avg_call_duration FLOAT,    -- niche benchmark
client_count INT,           -- "tested on 15 dental offices"
last_updated TIMESTAMPTZ
```

### How It Feeds Onboarding
When a new dental client signs up:
1. `NICHE_DEFAULTS['dental']` provides static structure (INDUSTRY, COMPLETION_FIELDS, etc.)
2. `niche_intelligence['dental']` provides DYNAMIC, battle-tested content:
   - "Here are the 12 questions dental callers actually ask (based on 2,000 calls across 15 offices)"
   - "This triage flow has a 78% completion rate — want to use it?"
   - "These 5 objection responses work best"
3. User sees: "Based on 15 dental offices on our platform, here's what works best for your agent"

### Privacy Model
- NEVER share specific client data across tenants
- Only share AGGREGATE PATTERNS:
  - Question frequency (not who asked)
  - Answer effectiveness (not which business answered)
  - Triage conversion rates (not which client's calls)
  - Objection frequency (not specific caller interactions)
- Each client's `knowledge_chunks` remain isolated (RLS enforced)
- Niche intelligence is computed from anonymized aggregates only

### The Flywheel
```
Client 1 onboards → handles calls → generates data
                                          ↓
                              Niche intelligence aggregation (cron)
                                          ↓
                              Better defaults for niche
                                          ↓
Client 2 onboards → better out of box → handles calls → more data
                                                              ↓
                                                  Even better defaults
                                                              ↓
Client N onboards → near-perfect out of box
```

### Implementation Approach
1. **Cron job** (weekly): aggregate knowledge_query_log + call_insights per niche
2. **Niche intelligence table**: store top questions, proven FAQ, best triage
3. **Onboarding integration**: `buildNicheFaqDefaults()` reads from niche_intelligence instead of static NICHE_DEFAULTS when sufficient data exists (minimum: 3 clients, 100 calls)
4. **Dashboard badge**: "Powered by insights from 15 dental offices" — trust signal
5. **Admin view**: niche intelligence dashboard showing aggregate stats

### Minimum Viable Version
Even before the full aggregation pipeline, we can do:
- Track which `buildNicheFaqDefaults()` entries actually get used vs removed by clients
- Track which FAQ answers match caller questions (knowledge_query_log hit rate)
- Manually update NICHE_DEFAULTS quarterly based on production data
