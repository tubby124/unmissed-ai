# D289 — Services Input UX (structured chips, not text box)

## Problem
Three disconnected systems for services:
1. **Onboarding:** free text box → parsed into `services_offered` string variable
2. **Dashboard ServiceCatalogCard:** structured items (name, duration, price) in `client_services` table
3. **Prompt SERVICES_OFFERED variable:** flat string in the template

User types "drain cleaning, water heaters, bathroom renos" into a box. That's the worst possible UX for structured data.

## Vision

### Onboarding Step
```
What services does Prairie Plumbing offer?

Pre-populated for plumbing:
[✅ Drain Cleaning] [✅ Water Heater] [✅ Burst Pipe Repair]
[✅ Sewer Backup] [✅ Bathroom Renovation] [✅ Kitchen Plumbing]
[  Sump Pump] [  Water Softener] [  Gas Line]

+ Add custom service: [____________] [Add]

Duration & pricing (optional — helps your agent):
| Service | Duration | Price range |
|---------|----------|-------------|
| Drain Cleaning | 1-2 hrs | $150-300 |
| Water Heater | 2-4 hrs | $800-2000 |
```

### Post-Onboarding (Dashboard)
Same component, always editable. Changes flow through D260 (service catalog sync) to the live agent.

## Data Source for Pre-Population
- `NICHE_DEFAULTS[niche].SERVICES_OFFERED` — current defaults as starting chips
- Niche-specific service lists (need to create structured arrays per niche)
- D246 extracted services from website scrape
- AI Compiler extracted services from uploaded docs

## Single Source of Truth
After this ships: `client_services` table is THE source. The flat `services_offered` string is derived from it. No more text box → parse → pray.
