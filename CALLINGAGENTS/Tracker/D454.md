# Treat clients.system_prompt as a derived cache

## Origin (2026-05-05)

The deepest cause of Brian's drift: `clients.system_prompt` is treated as both source-of-truth AND derived artifact, depending on which code path you ask. Anything (dashboard, scripts, manual SQL, `/build-prompt-ui`) can write loose text directly. Slot pipeline assumes the column is a regenerable cache; manual edits assume it's authoritative. The contract is broken.

Once D453 (structured overrides) exists, the column has no legitimate reason to be edited directly — every change should flow through `recomposePrompt()` which reads structured DB columns.

## Build

### Phase A — Audit

Enumerate every direct UPDATE to `clients.system_prompt`:
- `PATCH /api/dashboard/settings` (via `applyPromptPatches`) — legitimate, slot-pipeline path
- `recomposePrompt()` in [src/lib/slot-regenerator.ts](src/lib/slot-regenerator.ts) — legitimate
- `activateClient()` provision route — legitimate (initial composition)
- `deploy_prompt.py` script — semi-legitimate (snowflake clients only)
- `prompt-deploy` skill (Python) — semi-legitimate
- Any direct SQL or admin-recompose paths — should route through `recomposePrompt()`

### Phase B — Allow-list enforcement

Two options, pick after Phase A:

**Option 1 — DB trigger:**
```sql
CREATE OR REPLACE FUNCTION block_direct_system_prompt_write()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  IF current_setting('app.allow_system_prompt_write', true) IS NULL THEN
    RAISE EXCEPTION 'Direct writes to clients.system_prompt are blocked. Use recomposePrompt() or set app.allow_system_prompt_write=true.';
  END IF;
  RETURN NEW;
END $$;
CREATE TRIGGER guard_system_prompt
BEFORE UPDATE OF system_prompt ON clients
FOR EACH ROW EXECUTE FUNCTION block_direct_system_prompt_write();
```
Allowed callers (`recomposePrompt`, `applyPromptPatches`, etc.) set the session var before write.

**Option 2 — Service-role-only RLS:**
RLS policy denies `system_prompt` updates from anon/authenticated; only service-role bypasses. All allowed paths already use service role. Less granular than Option 1 but simpler.

### Phase C — Migration safety

Before enabling: run a 1-week soak with the trigger in **warning-only mode** (logs to `system_prompt_write_audit` table but doesn't block). Capture every caller. Add any missed legitimate ones to the allow-list. Then flip to enforce.

## Acceptance

- [ ] All legitimate write paths set the session var / use service role
- [ ] Test: direct `UPDATE clients SET system_prompt = '...'` from dashboard auth fails
- [ ] Test: `recomposePrompt()` succeeds
- [ ] 1-week soak audit shows zero unaccounted write attempts

## Dependency order

**BLOCKED on D453.** Don't enforce write-protection until structured overrides exist as the legitimate edit path — otherwise admins lose their "edit prompt" capability with no replacement.

## Why this is the structural fix

D451 + D452 + D453 reduce the probability of drift. D454 makes drift impossible by construction. Without D454, every new entry point (new admin tool, new skill, new migration script) reintroduces the risk. With D454, drift requires explicitly bypassing the guard — a clear violation, easy to catch in review.
