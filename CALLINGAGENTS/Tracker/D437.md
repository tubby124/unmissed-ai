# D437 — Concierge Provisioning Completion Gate + Pricing Model Pivot

## Status
OPEN — P0 study task. No code yet. Triggered by Velly real-world data showing the gap.

## Problem (the "Did Kausar actually finish?" gap)

When we hand-provision a concierge client, we have **no automated way to know whether they completed their side of the setup**. Each completion gate is a separate Supabase column, separate API surface, separate Stripe state. We currently figure it out by running 5 manual SQL queries.

### Real-world evidence — Velly Remodeling (Kausar Imam) as of 2026-04-29 ~00:20 UTC

| Gate | DB field | State | Reality |
|---|---|---|---|
| Account created | `auth.users.id` | ✅ 2026-04-28 23:49 | Exists |
| Email confirmed | `auth.users.email_confirmed_at` | ✅ 2026-04-28 23:49 | Clicked welcome email |
| **Dashboard login** | `auth.users.last_sign_in_at` | ❌ NULL | **Never signed in** |
| **Telegram linked** | `clients.telegram_chat_id` | ❌ NULL | **Deep link not tapped** |
| **Card on file** | `clients.stripe_customer_id` | ❌ NULL | **No payment method** |
| Forwarding active | First non-Hasan inbound `call_logs` | ❌ none yet | Carrier code not dialed |
| Minutes used | `clients.seconds_used_this_month` | 172s (~2%) | Tests only |

Hasan **believed** Kausar's Telegram was set up. Data says no. Likely path: tapped magic-link in email on phone (confirmed email but session didn't carry to dashboard), downloaded the Telegram *app* but never tapped the `t.me/hassitant_1bot?start=<token>` deep link.

This is exactly the "did they finish onboarding" funnel hole that kills concierge conversion.

## Two questions this task must answer

### Question 1 — Build a Provisioning Completion Gate
**Goal:** one command that tells us where any concierge client is in the funnel + auto-recommends the next nudge.

Proposed: `/concierge-status [slug]` skill + a Supabase view `client_provisioning_state` that derives:
- `account_exists` (bool)
- `email_confirmed` (bool)
- `dashboard_logged_in` (bool — derived from `auth.users.last_sign_in_at`)
- `telegram_linked` (bool — derived from `clients.telegram_chat_id`)
- `card_on_file` (bool — derived from `clients.stripe_customer_id`)
- `forwarding_activated` (bool — derived from first inbound `call_logs.caller_phone NOT IN owner_phones`)
- `gate_stuck_at` (string — name of the first failing gate)
- `hours_since_provision` (numeric — for nudge timing)

Then suggest the right next action based on `gate_stuck_at` + time elapsed:
- No login >24h after provision → re-send magic link
- No Telegram tap >24h → SMS the deep link directly
- No card >48h → re-send Stripe Payment Link as standalone reminder
- Card added but no inbound calls >72h → carrier-forwarding nudge (most common drop-off)

Same pattern as `/blitz-status` for True Color leads.

### Question 2 — Pricing Pivot: Free 50 min vs Card-Required 50 min Bonus

This is the **bigger** question. Concierge provisioning takes 1-2 hours of Hasan's time per client. We can't afford that for prospects who won't even tap a Telegram link.

| Model | Funnel width | Conversion to paid | Tire-kicker rate | Best fit |
|---|---|---|---|---|
| Free 50 min, no card (current Lite plan path) | Wide | ~8-15% (industry) | High | Self-serve SaaS at scale |
| **Card required, 50 min "trial credit"** | Narrow | **70-90%** (Stripe data) | Near-zero | **Concierge / high-touch** |
| Card required, charge immediately | Narrowest | n/a (already paid) | Zero | What Velly is doing now |

### Recommendation (to validate during study)

For the concierge model **as currently practiced** (manual provision per client, $29 Core, Hasan-led):
- **Lock in card-required-with-50-min-bonus** as the default for next concierge client
- Use Stripe `trial_period_days` with payment method required
- Charge fires automatically on day N if not cancelled
- Friction filters serious leads — Hasan's time is the scarce resource

Where this might not hold:
- If we ever shift to self-serve onboarding (D291 GBP auto-import, D293 paste-URL flow), free trial without card might be necessary again to widen the funnel
- For "founding member" deals (Velly-style), pre-charging is fine and we already do that
- Decision should NOT change Velly mid-flight — let her existing $29 setup play out

### Phase B — Build the Gate (after pricing decision)
1. **Supabase view** `client_provisioning_state` — derives all 6 gate states
2. **Skill** `/concierge-status [slug]` — outputs gate table + recommended next nudge
3. **Dashboard widget** (admin-only at first) — shows gate state for each client, aging counters, one-click re-nudge actions
4. **Nudge templates** — pre-written email + SMS for each stuck-gate state, sendable via gmail.py + Twilio

### Phase C — Pricing Pivot (if decision is card-required)
1. Build Stripe Checkout session with `payment_method_collection=always` and `trial_period_days=N`
2. Update `/onboard-client` skill to default to card-required path
3. Update Concierge-Onboarding-SOP to reflect new default
4. Build lite-tier escape hatch ONLY for prospects who explicitly refuse card (rare, manual approval)
