# S16a — Call Recording Consent Disclosure

## Status
2026-04-28 — Universal consent shipped. Onboarding now requires a checkbox before launch. New clients auto-enable RECORDING_DISCLOSURE. The 4 grandfathered clients (hasan-sharif, exp-realty, windshield-hub, urban-vibe) get a one-time login modal before they can use the dashboard — but their prompts are NOT auto-modified (they opt-in to the disclosure separately to avoid an unauthorized prompt redeploy).

Outbound calling (currently `dashboard/leads/dial-out`) is hard-gated on `recording_consent_acknowledged_at IS NOT NULL` — fail-closed 403 with clear error.

Voicemail pipeline (`prompt-niches/voicemail-prompt.ts`) now mirrors the slot-pipeline disclosure render — closed the prior gap.

See [[Features/Recording-Consent]] for the full design.

---

## Original mechanism (still applies — now flipped from opt-in-by-default to opt-in-on-acceptance)
Mechanism shipped 2026-04-22. Default is OPT-IN (empty string) — does not break existing clients or 1683 golden tests.

## Why opt-in default (read this before touching)
Every inbound call path already has `recordingEnabled: true` in Ultravox (verified in webhook-security-and-idempotency.md §1). So consent disclosure is a **compliance gate**, not a runtime toggle.

Initial approach tried a non-empty default so every new prompt would include the disclosure line. This regenerated the 5 Layer 1 snapshot fixtures AND surfaced a pre-existing stale-snapshot issue on `main` (auto-glass-baseline has transfer=enabled in the committed .txt but `buildPromptFromIntake` with the test fixture produces transfer=disabled). That rabbit hole ate a session.

**Rule:** Any future change to `buildSlotContext` or `buildConversationFlow` must either (a) keep output byte-identical for the Layer 1 fixtures, or (b) regenerate snapshots AND investigate the auto-glass transfer delta separately. Do not bundle them.

## What shipped
- `src/lib/prompt-slots.ts` — `SlotContext.recordingDisclosure: string` + render after greeting in both the `message_only` path and the main `buildConversationFlow` path. When empty (default), the greeting block is unchanged.
- Override path: set `niche_custom_variables.RECORDING_DISCLOSURE` on the client, or bake into a niche default map.
- Canonical suggested value: `"and heads up — this call's being recorded for quality."`

## What is NOT done (and why)
- Not enabled on any of the 4 live clients (hasan-sharif, windshield-hub, urban-vibe, exp-realty) — standing rule bans redeployment to those.
- Not enabled by default for new concierge clients yet — needs a one-line niche-default update once the concierge SOP is validated (D380).
- Voicemail path (`prompt-niches/voicemail-prompt.ts`) not modified — that pipeline has its own OPENING block; add there separately when turning S16a fully on.

## How to turn it on for a new client
```sql
update clients
set niche_custom_variables =
  jsonb_set(
    coalesce(niche_custom_variables, '{}'::jsonb),
    '{RECORDING_DISCLOSURE}',
    '"and heads up — this call''s being recorded for quality."'
  )
where slug = '<new-client-slug>';
```
Then trigger `recomposePrompt()` (D307 safety gate will show diff).

## Jurisdiction note
- One-party consent (most of Canada + many US states): the disclosure alone satisfies notice requirements.
- Two-party consent (California, Florida, some others): the disclosure does NOT substitute for explicit yes/no consent. For those clients, extend the prompt to ASK — out of scope for S16a.
