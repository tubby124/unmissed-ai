# D445 — Migrate 4 legacy-monolithic snowflakes to slot-pipeline format

> **PRE-CONDITION (HARD BLOCKER):** `recomposePrompt()` at [slot-regenerator.ts](../../src/lib/slot-regenerator.ts#L539-L540) has a `hasSlotMarkers()` guard that rejects prompts without slot markers — i.e., it refuses to run on exactly the prompts this migration targets. The audit's recommended `recomposePrompt({ slug, dryRun: true })` workflow returns `success:false` for every snowflake. Migration cannot proceed until either (a) a `forceRecompose: true` bypass flag is added or (b) a one-shot script calls `buildPromptFromSlots(ctx)` directly, bypassing the guard. **Design + ship this bypass first.** Surfaced by [[Architecture/Snowflake-Migration-Deep-Plan]] addendum 1, 2026-04-30.

> **Sequence updated 2026-04-30:** Deep plan inverts the audit's call-volume-first ordering. Migrate `windshield-hub` (LOW risk, simplest prompt, fewest features) → `urban-vibe` (MED) → `exp-realty` (HIGH) → `hasan-sharif` (HIGH, most custom phrasing + owner edit frequency). Smallest blast radius first to prove the procedure; highest-risk last with all learnings applied. See [[Architecture/Snowflake-Migration-Deep-Plan]] for per-client risk rationale.

> **Reference:** Full procedure, rollback, acceptance criteria → [[Architecture/Snowflake-Migration-Deep-Plan]] (40KB draft).

## Why this replaces the original D442 plan's "Fix 5"

The plan scoped the migration to **Urban Vibe alone** on the assumption it was uniquely broken. The D442 Phase 1 audit (2026-04-30, read-only, all 5 clients) found that **all 4 snowflakes** — `hasan-sharif`, `exp-realty`, `urban-vibe`, `windshield-hub` — are equally broken at the slot level: zero `<!-- unmissed:* -->` markers, identical `regenerateSlots` failure mode (`{ok:true, warning:'Old-format prompt without section markers — use patchers instead of regeneration', promptRegenerated:false}`).

Urban Vibe is **not special on impact**. It is 1 of 4 equally-broken legacy clients.

## Recommended sequence (if shipped at all — see open question below)

`hasan-sharif` → `exp-realty` → `urban-vibe` → `windshield-hub`.

Sequence by call volume / owner edit frequency, not by who's complaining. Hasan is the engineer-owner so highest ROI; Omar/Fatema's exp-realty is second-highest impact; urban-vibe (Ray) third; windshield-hub (Mark) fourth (most stable, fewest surfaces — no transfer, no booking).

## Per-client procedure (each migration is its own atomic ticket: `D445-{slug}`)

1. Verify snapshot exists at `docs/refactor-baseline/snapshots/2026-04-30-pre-d442/{slug}-system-prompt.txt`.
2. Run `recomposePrompt({ slug, dryRun: true })` and diff against current `system_prompt`.
3. Owner approval (Hasan) for the diff — review what changes, confirm intended.
4. `validatePrompt()` on new prompt — must be < 12K chars (per `prompt-edit-safety.md`).
5. `/prompt-deploy {slug}` — **lifts the standing no-redeploy rule for this slug for this deploy only** (not blanket).
6. Run a test call → `/review-call {ultravox-call-id}` → confirm greeting + AGENT_NAME + business name match expected.
7. Verify `last_agent_sync_status='success'` post-deploy.
8. Mark `D445-{slug}` done; **do NOT cascade to next snowflake until owner re-approves**.

## Open question — should we migrate at all?

Both the D442 plan and the audit reach the same conclusion:

> If Fix 1 + 1.5 + 2 ship cleanly, the user-facing trust gap is fixed by surfacing runtime truth + per-field warnings. The snowflake migration becomes optional polish, not a blocker.

**Recommended decision criterion:** if any of the 4 snowflake owners (Hasan, Omar/Fatema, Ray, Mark) hits a per-field "Saved, but not live yet" warning AND requests their prompt edit propagate, migrate that one client. Otherwise defer indefinitely. This is the same pragmatic posture D304 settled on (`wont-fix` 2026-04-06) — but with the audit data behind it now.

## Standing rules (non-skippable)

- The standing **no-redeploy rule** (per `.claude/rules/refactor-phase-tracker.md` and `core-operating-mode.md`) applies until owner explicitly lifts it per client. Each migration is one explicit lift.
- The full system_prompt snapshot in `docs/refactor-baseline/snapshots/2026-04-30-pre-d442/` is the rollback artifact. Reverting = paste snapshot back to `clients.system_prompt` + `updateAgent()`.
- **Pre-deploy:** take a fresh snapshot named `{slug}-system-prompt-pre-d445-{date}.txt` immediately before the deploy, in case settings have changed since the audit. The 2026-04-30 snapshot stays as the audit baseline.
- D304 was marked `wont-fix` on 2026-04-06 with the rationale that the founding 4 are permanently frozen. D445 reopens this **only** under the per-client owner-lift gate above. If owner declines lift, D445-{slug} closes as `wont-fix` for that slug.
