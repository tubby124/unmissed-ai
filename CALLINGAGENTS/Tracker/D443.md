# D443 — Fix 1.5: Reject PATCH on registry-readonly variables; hide Edit button on UI

## Bug class
**fake-control (universal)** — affects all 5 clients regardless of pipeline (legacy-monolithic vs slot-pipeline).

## Problem
`PROMPT_VARIABLE_REGISTRY[GREETING_LINE].editable=false` is declared in the registry, but enforcement is missing system-wide:

1. The PATCH route (`src/app/api/dashboard/variables/route.ts`) does not check `varDef.editable` before accepting the write.
2. The Overview UI (`src/components/dashboard/home/AgentIdentityCard.tsx` `renderField`) renders the Edit button regardless of `varDef.editable`.

The contract is silently violated by both ends. On slot-pipeline clients (Brian's), the edit appears to work — but is contractually invalid. On legacy-monolithic clients, the edit silently no-ops in the prompt while the DB column updates. Either way, the surface is untrusted, which is exactly what Brian's voice memo flagged.

## Solution

### Backend — `src/app/api/dashboard/variables/route.ts`
PATCH handler. **After** the `getVariable(variableKey)` lookup, before any DB write:

```ts
if (!varDef.editable) {
  return NextResponse.json(
    { error: `Variable ${variableKey} is read-only` },
    { status: 400 }
  )
}
```

### Frontend — `src/components/dashboard/home/AgentIdentityCard.tsx`
`renderField` (~L391-L464). Gate the Edit button on `varDef.editable`. Look up `meta` from the variables GET response (already returned). Hide the Edit button when `editable === false`. Keep the value displayed read-only.

## Why this is one ticket and not five client-specific tickets
`editable=false` is declared in the registry but enforcement is missing system-wide. Closing both ends (backend reject + frontend hide) closes the universal fake-control across all clients regardless of slot vs legacy pipeline. No per-client work needed.
