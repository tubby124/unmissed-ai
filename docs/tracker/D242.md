# D242 — niche='other' → closest production template

## Problem
When a niche slug doesn't match a production key, the agent falls through to the sparse `NICHE_DEFAULTS.other` template instead of a production-quality one.

## What Was Done (2026-04-04)
Implemented `resolveProductionNiche(niche: string): ProductionNiche` in `src/lib/prompt-config/niche-defaults.ts`.

Keyword mapping:
- trades / auto glass / HVAC / plumbing / mechanic / painting / pest / cleaning → `auto_glass`
- property / rental / strata / leasing / landlord / tenant → `property_management`
- sales / leads / ISA / real estate / realtor / outbound → `outbound_isa_realtor`
- anything else → `auto_glass` (most complete template, best default)

## Files Changed
- `src/lib/prompt-config/niche-defaults.ts` — added `resolveProductionNiche()` at bottom
- `src/lib/prompt-slots.ts` — `buildSlotContext()` now calls `resolveProductionNiche(niche)`
- `src/lib/sms-template.ts` — `buildSmsTemplate()` now calls `resolveProductionNiche(niche)`

## Known Gap
Original ticket described a Haiku inference + UI confirm step (dynamic). We shipped static keyword mapping instead — simpler, no LLM latency on the critical path. The UI confirm step was deferred.

`admin/preview-prompt/route.ts` and the legacy `buildPromptFromIntake()` path in `prompt-builder.ts` do NOT yet use `resolveProductionNiche`. Those are separate cleanup items.
