# D422 — PM post-onboarding editable fields panel (maintenance contacts, pet policy, tenant roster)

**Status:** 🔴 Open
**Priority:** HIGH
**Date:** 2026-04-05

## Problem
The chip fields a PM filled during onboarding (pet policy, parking, packages, maintenance contacts, tenant roster) are baked into the system prompt at provision time. After onboarding, there's NO UI to edit them. If a PM hires a new plumber, they have to:
1. Open Settings → find Prompt Editor
2. Edit the raw system prompt text
3. Hit "Sync Agent"

That's a terrible workflow. No PM will do this. Their maintenance contact list goes stale and they blame the product.

The tenant roster goes into `context_data` (editable via AdvancedContextCard), but it's labeled "TENANTS" in a raw textarea — not a clear edit surface.

## Solution
Add a **PM Configuration** card to settings, visible only when `niche === 'property_management'`. This card surfaces ALL onboarding chip answers as post-onboarding editable fields:

- Pet policy (chip selector, same options as onboarding)
- Parking (chip selector)
- Package handling (chip selector)
- Maintenance contacts (textarea — name + phone + what they handle)
- Tenant roster (table or textarea — unit + name + rent)
- Emergency tech phone (phone input — same as after_hours_emergency_phone)

On save: update `niche_custom_variables` or a dedicated `niche_answers` column → trigger slot regen → sync agent.

This means every PM setting is editable from the dashboard, not just accessible during the one-time onboarding flow.

## Files
- `src/components/dashboard/settings/PmConfigCard.tsx` (new)
- `src/components/dashboard/settings/AgentTab.tsx` — render PmConfigCard when PM niche
- `src/lib/settings-schema.ts` — if new fields needed (or reuse niche_custom_variables)
