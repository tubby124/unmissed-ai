# D304 — Old-Client Prompt Migration (Add Section Markers to 4 Live Clients)

## Problem
The 4 live clients (hasan-sharif, exp-realty, windshield-hub, urban-vibe) have old-format prompts WITHOUT section markers (`<!-- SLOT:xxx -->`). The slot regenerator has an `hasSlotMarkers()` guard that correctly refuses to touch these prompts.

This means:
- Regeneration can't help them
- Variable edits still rely on fragile regex patchers
- They'll never benefit from Phase 6 recomposePrompt()
- Two code paths (patchers + regenerator) must be maintained indefinitely

## Fix Options

### Option A: One-shot migration script (RECOMMENDED)
- For each live client: read current `system_prompt` from DB
- Run `recomposePrompt()` (Phase 6) from their current DB state → produces slot-based prompt
- Diff against current prompt → human review
- If equivalent: write new prompt + add markers
- If different: flag for manual resolution

### Option B: Gradual marker injection
- When a patcher runs, inject section markers around the patched section
- Over time, all sections get markers
- Risk: incomplete coverage, partial marker state

### Option C: Leave them alone
- Keep patchers for old clients forever
- New clients get regenerator
- Accept dual maintenance cost

## Risk
HIGH — these are paying production clients. Any migration must be human-reviewed.
- Test with e2e-test-plumbing-co first
- Test call before/after to verify agent behavior unchanged
- Keep rollback ready (prompt_versions table)

## Depends On
- Phase 6 `recomposePrompt()` (D280) — the tool that produces the migrated prompt
- D302 — niche intake preservation (so migration is faithful)

## Decision (2026-04-06) — WONT FIX

Founding 4 clients (hasan-sharif, exp-realty, urban-vibe, windshield-hub) are permanently frozen. Their hand-crafted prompts will not be migrated. D304 is closed.

If their settings need changes: direct DB edit or manual prompt edit only. Never automated slot regeneration on these clients.
