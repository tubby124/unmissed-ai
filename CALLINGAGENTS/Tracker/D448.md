# D448 — Investigate `clients.tools` vs `Ultravox.selectedTools` divergence (universal `hangUp` gap)

> **This is a blocker for [[Tracker/D447]] tool-comparison logic.** D447 binds the Overview capability pills to runtime truth and uses `clients.tools` as the runtime-authoritative source per [[Architecture/Control-Plane-Mutation-Contract]] §7 risk #7 (`toolOverrides: { removeAll: true, add: client.tools }`). If that contract is **wrong** — i.e., the stored Ultravox `callTemplate.selectedTools` actually fall through to live calls — every divergence chip D447 renders is unreliable. Resolve D448 before D447 ships.

## What the D442 audit found

All 5 active clients have `hangUp` present in their Ultravox `callTemplate.selectedTools` but **absent** from `clients.tools`:

| Client | `clients.tools` count | UV `selectedTools` count | `hangUp` in DB? | `hangUp` in UV? | Other DB gaps |
|---|---|---|---|---|---|
| `hasan-sharif` | 5 | 7 | NO | YES | also missing `bookAppointment`, `checkCalendarAvailability`; carries dead `transitionToBookingStage` |
| `exp-realty` | 6 | 7 | NO | YES | — |
| `urban-vibe` | 5 | 5 | NO | YES | DB has `pageOwner`, UV has `hangUp` (bidirectional) |
| `windshield-hub` | 3 | 4 | NO | YES | — |
| `calgary-property-leasing` | 4 | 5 | NO | YES | — |

Per `buildAgentTools()` ([src/lib/ultravox.ts:786](../../src/lib/ultravox.ts#L786)), `hangUp` is the **only** "always" tool — registered for every client regardless of plan or capability. Yet none of the 5 DB rows include it.

**The contradiction:** if `toolOverrides: { removeAll: true, add: client.tools }` is what the inbound webhook actually sends, every production call would strip `hangUp` and the agents would not be able to end calls. **They do end calls correctly.** Something in the documented contract is wrong.

## Three competing hypotheses (audit §270-274)

| # | Hypothesis | Implication if true |
|---|---|---|
| H1 | `hangUp` is synthesized at call creation time **outside** `buildAgentTools()` and added to `toolOverrides` directly | Documented contract is incomplete — there's a hidden tool source. D447 must read both DB tools AND the synthesized list. |
| H2 | `removeAll: true` does **not** actually strip stored template tools the way [[Architecture/Control-Plane-Mutation-Contract]] §7 #7 documents | Documented contract is wrong. Stored Ultravox `selectedTools` are runtime-authoritative. `clients.tools` is decorative. D447 should bind to UV state, not DB. |
| H3 | `clients.tools` is **stale** and the next `syncClientTools()` will write `hangUp` correctly | Latent drift. Every settings PATCH that triggers `needsAgentSync` is currently writing tools arrays missing `hangUp`. Next sync would expose this. |

H2 and H3 are mutually exclusive on outcome but not on cause — H3 could be why H2 looks true (stale data masking the override). H1 is independent of both.

## Investigation procedure (30 min, focused)

Run [truth-tracer agent](../../.claude/agents/truth-tracer.md) — but here's the procedure for whoever picks it up:

### Step 1 — read the inbound webhook tool assembly path

[src/app/api/webhook/[slug]/inbound/route.ts](../../src/app/api/webhook/%5Bslug%5D/inbound/route.ts) — find every place the call's tool list is assembled. Specifically:
- The `callViaAgent()` invocation — what does the `overrideTools` argument contain?
- Is `client.tools` used directly, or is there a `[...client.tools, hangUpToolDef]` pattern hiding in the call site?

### Step 2 — read `callViaAgent` itself

[src/lib/ultravox.ts](../../src/lib/ultravox.ts) — `callViaAgent()`. Inspect the body sent to Ultravox `/api/agents/{id}/calls`. If `toolOverrides.add` is `client.tools`, H1 is false; if it's `[...client.tools, hangUp]` or similar, H1 is true.

### Step 3 — confirm what `removeAll: true` actually does

Two options:
- **Static:** read Ultravox docs (they are authoritative on this — the contract derived this from a doc reading, may have misread).
- **Empirical:** make a single live test call to e.g. `windshield-hub` with `clients.tools = [...current, plus a deliberately-removable test tool]`, confirm via call recording / agent transcript that `removeAll` actually behaves the documented way. Risk: live client. Use a synthetic test client (`e2e-test-plumbing-co`) instead.

### Step 4 — write a one-shot script that logs the **actual** tools array sent to Ultravox

Add a `console.log('[d448-trace]', JSON.stringify(toolOverrides, null, 2))` in `callViaAgent()` for one deploy, make a test call to each of the 5 clients (or just one — they share a code path), pull the Railway log, compare to `clients.tools`. Remove the log after.

### Step 5 — classify the finding

- If H1 is correct → document the synthesized tool source in [[Architecture/Control-Plane-Mutation-Contract]] §7 #7 and update D447 to read from both sources.
- If H2 is correct → rewrite [[Architecture/Control-Plane-Mutation-Contract]] §7 #7 entirely. Update D447 to bind capability pills to UV `selectedTools`, not DB `clients.tools`. The drift-detector agent rule also needs updating ([drift-detection-pattern.md](../../.claude/rules/drift-detection-pattern.md) currently treats `clients.tools` as runtime-authoritative).
- If H3 is correct → run `syncClientTools()` on all 5 clients (one PATCH per client to a benign field that triggers `needsAgentSync`). Confirm `hangUp` lands in `clients.tools`. Then either D447 reads the freshly-synced array, or D447 still needs to handle the next-stale-period gracefully.

## Why this matters beyond `hangUp`

`hangUp` is the canary because it's the ONE always-on tool. The same drift class exists for **any** tool that's been added to `buildAgentTools()` since the last `syncClientTools()` run on a given client. Three of the five clients have `last_agent_sync_status = 'unknown' / null` ([D442 audit table](../00-Inbox/overview-drift-audit-2026-04-30.md#L21-L25)) — meaning no recent settings PATCH triggered a re-sync. Any tool added in the last few months may be missing from those rows even if it ships correctly via the Ultravox stored template.

This contradicts the standing rule from [.claude/rules/command-routing.md](../../.claude/rules/command-routing.md):

> **`clients.tools` is the runtime-authoritative source — the agent's stored `selectedTools` is overridden by `toolOverrides` on every production call.**

If that rule is wrong, every `truth-tracer` and `drift-detector` invocation that anchored on it produced misleading results. Worth knowing before [[Tracker/D442]] Phase 2 work begins.

## Acceptance criteria

1. One of H1/H2/H3 is confirmed (or a 4th hypothesis is documented and confirmed).
2. [[Architecture/Control-Plane-Mutation-Contract]] §7 risk #7 + Section 4 ("Assuming tools are set on the agent at call time") are updated to match reality.
3. [.claude/rules/command-routing.md](../../.claude/rules/command-routing.md) "Coding patterns" section is updated if the `clients.tools` runtime-authority claim is wrong.
4. [[Tracker/D447]] divergence classifier knows which array to compare against and how (single source of truth or both).
5. If H3: confirm or schedule a full `syncClientTools()` sweep across the 5 clients. Standing no-redeploy rule still applies — this would be a tools-only update, no prompt change, but it does invoke `updateAgent()` which writes the prompt back too. **Verify the sweep is genuinely tools-only before running.**

## Risks

1. **Live test calls to confirm `removeAll: true` behavior** — use the synthetic test client, not a paying one.
2. **Adding the `console.log` trace** — counts as a deploy. One commit, deploy, run 5 test calls, revert. Don't leave a trace log in production.
3. **If H2 is true and the contract has been wrong since the original mutation-contract write**, then *every* prior tool-related drift fix and audit may have been chasing the wrong array. Don't retroactively re-investigate; just update the doc and move on.
4. **`syncClientTools()` writes `clients.tools` AND calls `updateAgent()`** ([src/lib/sync-client-tools.ts](../../src/lib/sync-client-tools.ts) — verify path before running). The latter rebuilds the Ultravox `callTemplate` including the prompt. Standing no-redeploy rule applies. If H3 is the answer, the sweep is a per-client owner-lift gate.

## Status

**resolved 2026-04-30 PM.** Contract is correct. The "universal `hangUp` gap" is an audit script extractor bug, not real drift.

### Findings

| Hypothesis | Outcome | Evidence |
|---|---|---|
| H1 (synthesized at call site) | **FALSE** | [src/app/api/webhook/[slug]/inbound/route.ts:322,348](../../src/app/api/webhook/%5Bslug%5D/inbound/route.ts) passes `client.tools` raw, no `[...client.tools, hangUpToolDef]` pattern. [src/lib/ultravox.ts:927](../../src/lib/ultravox.ts) — `body.toolOverrides = { removeAll: true, add: overrideTools }` with `overrideTools = client.tools` exact. |
| H2 (`removeAll: true` doesn't strip stored tools) | **FALSE** | Ultravox docs (`docs.ultravox.ai/api-reference/agents/agents-calls-post`) confirm `removeAll: true` removes all existing tools, no built-in exemption. Mutation contract §7 #7 was correct as written. |
| H3 (stale `clients.tools`) | **REFUTED on the symptom; partially correct in spirit** | Empirical Supabase diff on `windshield-hub` and `hasan-sharif` ([sub-agent dispatch 2026-04-30 PM](../00-Inbox/d448-empirical-diff-2026-04-30.md)): **`hangUp` IS present in `clients.tools` for both clients.** It uses key `toolName: 'hangUp'` (built-in tool format per [src/lib/ultravox.ts:125-130](../../src/lib/ultravox.ts)), NOT `modelToolName` (HTTP tool format). The D442 audit script's tool extractor likely scanned only `modelToolName` and missed all built-in tools. Audit table at `CALLINGAGENTS/00-Inbox/overview-drift-audit-2026-04-30.md:268-274` is wrong. |

### Real drift on the 5 clients (after corrected scan)

- **`windshield-hub`:** zero drift. Computed `buildAgentTools()` output = DB `clients.tools` exactly (4 tools).
- **`hasan-sharif`:** missing `pageOwner` in DB only. Added 2026-03-29 (commit `dc871e5`); client hasn't had a `needsAgentSync` settings PATCH since. **Single-tool drift, not the 3-tool gap the audit reported.**
- **`hasan-sharif` "missing `bookAppointment` + `checkCalendarAvailability`":** misread of call-stage architecture. `buildCalendarTools()` ([src/lib/ultravox.ts:420-422](../../src/lib/ultravox.ts)) returns ONLY `transitionToBookingStage`. The other two are stage-2 tools in `buildCalendarBookingTools()` ([line 337](../../src/lib/ultravox.ts)) used after the agent transitions into the booking stage. They are correctly absent from the main tool list.
- **`hasan-sharif` "dead `transitionToBookingStage`":** not dead. It IS the live output of `buildCalendarTools()`. Audit got this backwards.
- **`exp-realty`, `urban-vibe`, `calgary-property-leasing`:** not yet re-verified with corrected scan. Likely also `pageOwner`-only drift on clients with `forwarding_number` set.

### What was wrong: this tracker's own acceptance criterion #5

D448 spec said `syncClientTools()` "calls `updateAgent()` which rebuilds the prompt — standing no-redeploy rule applies." **This is wrong.** [src/lib/sync-client-tools.ts:1-53](../../src/lib/sync-client-tools.ts) is a pure DB write — no `updateAgent()` call, no Ultravox API call, no prompt touch. Header comment: *"Lightweight — no Ultravox API call. The runtime uses clients.tools via toolOverrides at call time."*

The H3 sweep is therefore safe to run without owner gating on prompt drift — it cannot rebuild the prompt because that code path doesn't exist. The standing no-redeploy rule does NOT apply to `syncClientTools()` as a standalone operation. It DOES apply to `settings/route.ts` PATCH path which does call `updateAgent()` for `needsAgentSync` fields.

### Unresolved gap (non-blocking for D447)

If `removeAll: true` truly strips everything and `clients.tools` lacks `hangUp`, agents shouldn't be able to end calls — but they do. Two candidates:
1. Ultravox's `removeAll` exempts platform-builtin tools (doc language ambiguous on this).
2. The Ultravox-stored `callTemplate.selectedTools` is somehow falling through despite the override.

Not investigated empirically (no test call made — D448 stayed read-only). **Not a blocker for D447** — D447 reads `clients.tools` as authoritative regardless. File as a separate observability ticket if it ever produces a real incident.
