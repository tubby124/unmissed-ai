# D429 — priorCallId: no returning caller memory

**Status:** open
**Priority:** P1
**Filed:** 2026-04-06

## Problem
Every call starts cold. A tenant who called 2 hours ago about flooding gets asked their name and unit number again. Ultravox supports `priorCallId` to inject prior call context into a new call, but it's not used anywhere in the codebase.

## Root Cause
`priorCallId` feature never implemented. `call_logs` table has the data needed (caller_phone, prior ultravox_call_id), but the inbound webhook doesn't look up prior calls or pass priorCallId to Ultravox.

## Fix
In `/api/webhook/[slug]/inbound/route.ts`, before creating the Ultravox call, query call_logs for the most recent call from the same caller_phone within the last 24h. If found, pass `priorCallId` to the Ultravox start-call API.

## Notes
This is the single biggest difference between "AI receptionist" and "annoying phone tree." High impact for PM niche (tenant callbacks on same issue).
