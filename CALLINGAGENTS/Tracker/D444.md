# D444 — Investigate `clients.tools.hangUp` missing on all 5 audited clients despite agents hanging up correctly

## Bug class
**path-parity OR source-of-truth contract violation** — needs investigation to narrow down.

## Symptom
The `clients.tools` array (per Section 7 risk #7 of the mutation contract, runtime-authoritative via `toolOverrides:{removeAll:true,add:client.tools}`) does **not** include `hangUp` on any of the 5 audited clients. Yet:

1. Every Ultravox `callTemplate.selectedTools` HAS `hangUp` deployed.
2. Production calls hang up correctly on all 5 clients.
3. `hangUp` is documented as the only "always" tool per `buildAgentTools()` in `src/lib/ultravox.ts` (line 786).

This contradicts the mutation contract's claim that `clients.tools` is the runtime source of truth.

## Three hypotheses (must be narrowed by truth-tracer or read of inbound webhook code)

1. **Synthesized at call creation time outside `buildAgentTools()`** — `hangUp` is pushed into `toolOverrides` directly by the inbound webhook or `callViaAgent()` wrapper, bypassing the DB array.
2. **`removeAll:true` doesn't actually strip stored template tools** the way the mutation contract documents — Ultravox may merge stored + override instead of replacing, in which case the agent's stored `selectedTools` (which has `hangUp`) fall through.
3. **`clients.tools` is stale** — predates the `hangUp` builder addition; next `syncClientTools()` would correct it. All 5 clients happen to be on the same stale-data vintage.

## Why this matters
All 5 clients tell us this is a **100% contract violation regardless of which hypothesis is correct**. It affects diagnostic confidence in `clients.tools` as a source of truth — which the `drift-detector` agent depends on per `.claude/rules/drift-detection-pattern.md`. If the mutation contract is wrong about `clients.tools`, then every drift audit going forward is reading the wrong source.

## Investigation plan (next agent)
1. Read `src/app/api/webhook/[slug]/inbound/route.ts` `callViaAgent()` invocation — does it pass a tool array that includes `hangUp` synthesized outside `clients.tools`?
2. Read `src/lib/ultravox.ts` `callViaAgent()` and the actual Ultravox call-creation POST body — confirm what `toolOverrides` looks like on the wire.
3. If hypothesis 1 confirmed: update Section 7 risk #7 to reflect that `clients.tools` is **almost** the runtime source — `hangUp` is synthesized separately. Add `hangUp` to either the DB array via a one-time backfill script OR document the carve-out explicitly.
4. If hypothesis 2 confirmed: file Ultravox-side bug + update mutation contract's Section 4 "Assuming tools are set on the agent at call time" to explain the actual merge behavior.
5. If hypothesis 3 confirmed: run `syncClientTools()` for all 5 clients to backfill `hangUp`.

## Investigation findings — 2026-04-30

**Hypothesis confirmed: NONE — the audit's premise is false. `clients.tools` DOES contain `hangUp` on all 5 clients. The drift report was generated by a faulty extraction script that did not recognize the built-in tool reference shape.**

**Evidence:**

1. Raw audit data — `CALLINGAGENTS/00-Inbox/d442-data/clients-raw.json` — every client's `clients.tools[0]` is `{ "toolName": "hangUp", "parameterOverrides": { "strict": true } }`. Counted by tool name: hasan-sharif=6 (incl. hangUp + dead `transitionToBookingStage`), exp-realty=7, urban-vibe=6, windshield-hub=4, calgary-property-leasing=5. The audit table reported 5/6/5/3/4 — those numbers are the result of skipping entries that have neither a `temporaryTool.modelToolName` key nor a top-level `toolName` key on the Ultravox side, which then was mistakenly applied symmetrically to the DB side.

2. Raw Ultravox data — `CALLINGAGENTS/00-Inbox/d442-data/ultravox/{slug}.json` — the deployed `callTemplate.selectedTools` references hangUp as `{ toolId: "56294126-5a7d-4948-b67d-3b7e13d55ea7", nameOverride: "hangUp", parameterOverrides: { strict: true }, descriptionOverride }`. Same identity, different wire shape (toolId reference vs name reference). The two shapes both bind to Ultravox's built-in `hangUp` tool — fully interchangeable.

3. `src/lib/ultravox.ts:124-130` — `HANGUP_TOOL = { toolName: 'hangUp', parameterOverrides: { strict: true } }`. This is the literal value `buildAgentTools()` writes when `tools` is undefined.

4. `src/lib/ultravox.ts:787` — `const baseTools: object[] = opts.tools !== undefined ? opts.tools : [HANGUP_TOOL]`. Single source. No second injection site.

5. `src/lib/sync-client-tools.ts:33-45` — calls `buildAgentTools()` without passing `tools`, so the `[HANGUP_TOOL]` default branch always fires. `clients.tools[0]` is therefore always `hangUp` after any sync run.

6. `src/app/api/webhook/[slug]/inbound/route.ts:322,348` — `const tools = client.tools` is passed as `overrideTools` unmodified. No append, no synthesis.

7. `src/lib/ultravox.ts:927` — `if (overrideTools?.length) body.toolOverrides = { removeAll: true, add: overrideTools }`. The hangUp already in `client.tools` is what flows through to Ultravox.

8. Git: commit `11fea5f` (2026-03-20) reverted `HANGUP_TOOL` to the built-in `{ toolName, parameterOverrides }` shape; commit `9cda553` (2026-03-21) extracted `buildAgentTools()` as the single source of truth; commit `4ff1ca6` (2026-03-25) is the latest functional change to `buildAgentTools` for SMS. From `9cda553` onward, every `syncClientTools()` run writes hangUp first.

**What this means for the contract:**

Section 7 risk #7 is **accurate as written**. `clients.tools` IS runtime-authoritative, `removeAll: true` DOES strip the stored agent's tools, and the contract's call-time merge model is correct. The drift report's hangUp finding was a tooling artifact, not a real divergence. No code change, no doc change, no backfill needed for the hangUp question.

The other findings in the D442 audit (e.g. `hasan-sharif.transitionToBookingStage` orphan, `urban-vibe.pageOwner` mismatch, the Greeting tile fake-control across all 5) are independent of this and remain valid — they don't depend on the hangUp extraction.

**Recommended action:**

(d) Close as "audit-script bug, no contract violation". Annotate the audit report with a correction note pointing here. Update D442 follow-ups to drop the "Universal investigation — hangUp missing on all 5" item. The drift-detector tool extraction script should be hardened to handle three shapes: built-in (`{toolName}`), referenced (`{toolId, nameOverride}`), and inline (`{temporaryTool.modelToolName}`).
